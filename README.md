# GLPI Operations Kit

This repository is a reusable infrastructure-and-operations kit for GLPI deployments on Ubuntu with Ansible, Nginx, PHP-FPM, and MariaDB.

Important: editing can happen on Windows, but operational execution must happen in Linux shell on target servers (or Linux jump host). Do not run deployment scripts from native Windows shell.

## Configuration model

Public baseline:

- `config/.env.example` (versioned template)
- `config/<environment>.env` (created by the operator from the template)

Secrets (never in Git):

- `config/<environment>.env` (operator-managed, not committed)
- `.runtime/<environment>/secrets.yml` (generated from environment config)

Runtime artifacts generated by scripts:

- `.runtime/<environment>/inventory.runtime.yml`
- `.runtime/<environment>/public.runtime.yml`
- `.runtime/<environment>/overrides.runtime.yml`
- `.runtime/<environment>/logs/`
- `.runtime/<environment>/state/`
- `.runtime/<environment>/evidence/`

## Supported topologies

- `dual-server`: one app host and one db host
- `single-server`: app and db on the same host

## Official CLI

```bash
./scripts/glpictl.sh <environment> <deploy|certify|promote|tls|ops|audit|email> <action> [target] [scope]
```

All scripts load `config/<environment>.env` automatically. Manual `export` is optional and only needed for one-off overrides.

## Quick start

```bash
cp config/.env.example config/staging.env
bash scripts/bootstrap-permissions.sh
./scripts/glpictl.sh staging deploy check all
```

Dual-server local flow without cross-host SSH:

1. On DB host, set `EXECUTION_HOST_ROLE_DEFAULT=db` in `config/staging.env`, then run:
   - `./scripts/glpictl.sh staging deploy apply db`
2. On APP host, set `EXECUTION_HOST_ROLE_DEFAULT=app` in `config/staging.env`, then run:
   - `./scripts/glpictl.sh staging deploy apply app`
   - `./scripts/glpictl.sh staging deploy apply monitoring`
   - `./scripts/glpictl.sh staging deploy apply backup`
   - `./scripts/glpictl.sh staging deploy post-check all`

## Capabilities

- permission bootstrap and precheck with remediation prompts
- single public config contract (`key=value`)
- runtime secrets isolation outside Git
- role-scoped deploy commands (`db`, `app`, `monitoring`, `backup`)
- TLS mode operations (`none`, `self_signed`, `provided`) with legacy and standardized aliases
- day-2 operations (`ops users`, `ops cert`, `ops audit`, `ops resume`)
- post-deploy Mailpit email capture (`email check|prepare|install|post-check|rollback mailpit`)
- readiness and evidence reports
- final execution summary with status, alerts, computed access URL, and test commands
- selectable policy mode per execution (`secure` or `permissive`)

## Authentication and SSO

SSO/SAML/OIDC configuration is manual in the GLPI application (and IdP). The kit does not orchestrate IntraID/Entra ID settings.

Legacy `AUTH_*` and `SSO_*` keys can still exist in old environment files and are ignored by execution flows.
